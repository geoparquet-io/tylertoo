## Findings & fixes

### F1 (FIXED) — duplicate / stale `bbox` covering on gpio inputs

The overview writer passed every source column through, then let the
geoparquet encoder *generate* its own `bbox` covering (§4.4). Every
gpio-optimized input (the documented input contract, §4.3) already
carries a `bbox` covering, so the output ended up with **two** `bbox`
columns. Worse, the `geo` covering metadata resolves the name `bbox`
to the *first* physical match — the passed-through **pre-generalization**
input bbox — so at coarse (simplified) levels the covering described the
*original* geometry, not the simplified one actually stored. (Douglas-
Peucker keeps a vertex subset, so the stale bbox is a conservative
superset — pruning stayed correct but not tight, and the fresh covering
was orphaned as `bbox_1`.) Datasets whose geometry column is named
`geometry` (all Overture / FTW) hit the name collision; Natural-Earth
(`geom` → covering `geom_bbox`) avoided the name clash but still carried
a redundant stale `bbox`.

**Fix** (`crates/core/src/overview/writer.rs`): the writer now drops any
pre-existing bbox-covering struct column whose name collides with the
covering it will generate, so the encoder's authoritative covering is the
only one present and the metadata points at it. Verified by unit test
`preexisting_bbox_covering_is_not_duplicated` and end-to-end here (no
`bbox_1` in any output; covering matches simplified geometry).

### F2 (OPEN, design) — `level` column name vs. source `LEVEL` attribute

The spec mandates a column named exactly `level` (§4.1). The writer only
rejects a source column named `level` **case-sensitively**, so a source
attribute differing only in case (Natural Earth admin `LEVEL`) passes
through and coexists with the overview `level`. Footer/parquet readers
(case-sensitive) are fine, but **DuckDB is case-insensitive**: `WHERE
level = <canonical>` binds to the *original* `LEVEL` attribute and
returns wrong/zero rows — silently defeating the spec's one-predicate
analysis contract (§5.3). Only `monster-admin` is affected in this corpus.
This is a design decision (error out? auto-rename the clashing source
column? spec guidance?), not a local writer bug, so it is documented
rather than fixed. `verify.sh` works around it by using DuckDB's
disambiguated `level_1` and flags the collision in the notes.

### F3 (INFORMATIONAL) — vertex-count non-monotonicity

For `polygons-ftw-moldova-large` total vertex count dips once
coarse→fine (level 7→8) even though feature count rises. This is a wide
feature-size distribution (coarse levels keep a few very large fields;
the next level adds many tiny ones), not a spec violation — §2.2 only
requires that a feature visible at level *k* appears at all finer levels
(feature-count monotonicity, which holds). Recorded as a note.
