# GeoParquet Overviews — Execution Plan

Status: draft, 2026-07-02
Goal: COG-style multi-resolution overviews embedded in GeoParquet,
generated by the existing gpq-tiles generalization engine.
Prior art context: Kanahiro COGP/yosegi (thinning + layout, no
simplification), ESRI Spatially Optimized Parquet (per-scale geometry
columns, proprietary), CARTO quadbin tilesets (cell aggregation).
Our position: row model, self-contained levels (COG semantics),
COGP-style layout (level-major, RG-aligned, spatial sort within level),
plain `level` column + footer metadata key. Files remain valid
GeoParquet 1.1 (2.0 variant later).

Execution model: agent-driven. Each task below is scoped to be a
single agent assignment. Tasks in the same phase with no arrow
between them run in parallel. TDD per repo convention; targeted
tests only (see CLAUDE.md).

## Phase 0 — Groundwork (all three in parallel, no dependencies)

### G1. Format design doc (`context/OVERVIEWS_SPEC.md`)
The single most load-bearing artifact; everything downstream reads it.
Must decide:
- Footer key name + JSON schema: version, levels[] with
  {row_group_end, gsd, zoom?, mode}, canonical level pointer,
  generalization params provenance.
- Level modes: `duplicating` (each level self-contained, COG read
  semantics: read exactly one level band) and `partitioning`
  (COGP-compatible prefix reads). v1 implements duplicating;
  spec accommodates both.
- `level` as a plain visible INT32 column (naive-reader filterability)
  AND footer row-group ranges (footer-only readers).
- GSD/zoom mapping (adopt cogp-rs convention:
  gsd(z) = 40,075,016.69 / 1024 / 2^z; degrees→meters factor for
  geographic CRS).
- Layout rules: levels coarse→fine, each level ends on a row-group
  boundary, Hilbert order within level (input contract: gpio-sorted;
  fallback sort optional later), bbox covering column per GeoParquet
  1.1, ZSTD, no dictionary encoding for geometry/bbox.
- Read protocol: display scale → GSD → level band ∩ bbox-pruned
  row groups; analysis convention: WHERE level = <canonical>.
- Antimeridian + CRS constraints (v1: follow COGP restrictions).
- Explicit non-goals for v1: topology preservation across features,
  in-place updates, multi-layer.
Output: spec doc + worked example (tiny file, annotated footer).
HUMAN CHECKPOINT: approve spec before Phase 1 merges anything.

### G2. Codebase audit + carry-over map
For each core module: works / broken / unknown, and its fate
(reuse / adapt / shelve). Expected shape:
- Reuse: simplify.rs (per-level tolerances), feature_drop.rs,
  sampling.rs, gap_density.rs, clustering.rs (later), wkb.rs,
  world_coord.rs, memory.rs, spatial_index.rs, tile.rs (virtual
  grid math only).
- Shelve (keep for serve-bridge, not in overview path): clip.rs,
  hierarchical_clip.rs, sutherland_hodgman.rs, ioverlay_clip.rs,
  dedup.rs, mvt.rs, pmtiles_writer.rs, coalesce.rs(?),
  accumulator.rs, external_sort.rs.
- Document *why* the tile pipeline is broken/stuck (test failures,
  known quality gaps) so we don't re-import the problem.
Output: `context/CARRYOVER.md`.

### G3. Test corpus + golden references
- Datasets (small/medium/large per class): points (POIs), lines
  (roads — COGP's worst case, our best case), polygons (buildings,
  parcels), monster geometries (coastline, admin boundaries).
  Prefer Overture extracts + one gpio-optimized copy of each.
- Golden references: tippecanoe output per zoom for the same inputs
  (visual + count baselines); cogp-rs output for thinning parity.
- Scripted fetch/build under `tests/fixtures/` or a data/ script
  (don't commit large files; commit the recipe).
Output: corpus script + manifest.

## Phase 1 — Prototype (in-memory OK; correctness > performance)

### P1. Level-assignment engine  (deps: G1, G2)
New module `overview/assign.rs`: cell-winner thinning per level.
- Grid per level from GSD; representative point per feature;
  one winner per cell; priority = (sort_key?, size desc, stable hash).
- Kind-aware factors (point/line/polygon thinning factors) and
  bbox-diagonal visibility gates, per cogp-rs.
- In duplicating mode output is: for each level, the SET of surviving
  features (feature may appear at many levels), plus per-feature
  min-level. Pure function over (features, level params) → assignments.
TDD: unit tests on synthetic grids; parity test vs cogp-rs thinning
on a fixture (partitioning mode).

### P2. Per-level simplification  (deps: G1, G2, simplification branch)
Adapt simplify.rs to world-space tolerance derived from level GSD
(not tile pixel space). Degenerate-geometry guards (collapse to
point/drop below visibility gate). Per-kind rules: points pass
through; lines/polygons simplify; rings must stay valid.
TDD: tolerance-per-level table tests; validity property tests.

### P3. Overview GeoParquet writer  (deps: G1)
New module `overview/writer.rs`:
- Input: per-level feature streams (from P1/P2).
- Writes levels coarse→fine; row-group alignment at level boundaries;
  bbox struct column + GeoParquet 1.1 `geo` metadata with covering;
  `level` column; footer key per spec; ZSTD.
- Preserve input column schema; canonical level bytes should be
  equivalent to input data (geometry unmodified).
TDD: footer round-trip; RG-boundary invariants; DuckDB can read it.

### P4. Reader + `validate` subcommand  (deps: P3)
`overview/reader.rs`: parse footer, select level by GSD/zoom,
bbox-prune row groups, stream batches (reuse geoparquet crate +
object_store). CLI `gpq-tiles validate <file>` checks spec
invariants (monotone GSDs, RG alignment, level column ↔ footer
consistency, covering stats present).
TDD: reader selects correct byte ranges on fixture files.

### P5. Converter CLI  (deps: P1, P2, P3)
`gpq-tiles overview input.parquet output.parquet
  [--min-zoom --max-zoom | --gsd ...] [--mode duplicating|partitioning]
  [--sort-key col] [--thinning/visibility factors]`
Wire pipeline: read (assume gpio-sorted) → assign → per-level
simplify → write. Python binding comes in E3.
Integration test: each corpus dataset converts, validates, and
DuckDB `WHERE level = canonical` row count == input row count.

## Phase 2 — Validation

### V1. Correctness suite  (deps: P5)
- File validity: GDAL ogrinfo, DuckDB spatial, QGIS open, pyarrow.
- Canonical-level fidelity: geometry + attributes byte/value-equal
  to input.
- Partitioning mode with `cogp` compat key enabled: readable by
  cogp-rs (run cogp validate if feasible; optional interop check).
- Determinism: same input → same output.
FORMAT FREEZE after V1 passes + G1 checkpoint feedback applied.

### V2. Quality evaluation  (deps: V1, G3)   [HUMAN GATE]
- Render every corpus dataset at each level (static PNGs via a
  small render script) side-by-side with tippecanoe's corresponding
  zoom renders.
- Automated metrics: features/level, vertices/level, level band
  sizes, density histograms; regression-tracked.
- HUMAN CHECKPOINT: you judge the renders. Outcome decides whether
  clustering/coalescing (V5, later) is required before launch or
  can follow it.

### V3. Benchmarks  (deps: V1, G3)
The publishable numbers; nobody has produced these:
- Storage: overview file vs plain gpio file vs gpio+PMTiles vs COGP.
- Access: bytes fetched + request count + wall time for
  world-view / regional / street-level viewports over HTTP
  (localhost + real S3/R2), vs PMTiles and vs COGP.
- Write: conversion time + peak RSS vs tippecanoe on same input.
Output: `benchmarks/overview/` + results doc with method notes.

### V4. Streaming refactor  (deps: V1; parallel with V2/V3)
Multi-pass bounded-memory pipeline:
pass 0 stats (optional) → pass 1 build cell-winner tables →
pass 2 per-level stream-simplify-write. Memory target:
O(row group + winner tables). Verify peak RSS on the large corpus
item stays under budget (dhat/heaptrack test).
(PRIORITY UP per V1 results 2026-07-02: Moldova 632k polygons =
12m04s / 5.5 GB peak RSS in the in-memory v1. Also revisit wall
time — likely dominated by per-level decode/rebuild.)

### V5. Quality ladder (REVISED 2026-07-02 after early V2 renders:
mid/fine zooms good; coarse-zoom line networks fragment. Waves
below; each ends with partial re-render + human eyeball gate.)

Dependency graph:
```
Q1 class priority ──┬─→ Q2 density budgets ──┐
                    └─→ Q3 line coalescing   ├─→ Q5 polygon coalescing
Q6 aggregation semantics ─→ Q4 point clustering ┘
```

- **Q1 (Wave 1, NOW): class-aware priority.** Ranking tiers:
  (1) user-named rank column / categorical mapping;
  (2) fallback: size (bbox diag), then deterministic hash
  (fair random — tippecanoe-like) — already implemented;
  (3) auto-detect well-known schemas (Overture roads `class`,
  Overture places `confidence`). Record ranking used in the
  footer `generalization` provenance block. Gate: Portland/Boise
  roads coarse renders.
- **Q2 (Wave 2): density-based budgets** — per-level feature
  budgets / drop-rates varying with local density (dense areas
  drop harder), consuming Q1's ranking. Reuse gap_density.rs /
  sampling.rs concepts. Gate: NYC 458k POIs at z4–z8.
  ∥ **Q6: aggregation semantics decision** (paper): attributes of
  merged features — tippecanoe accumulate-attribute conventions,
  accumulator.rs; spec v0.2 note. Prereq for Q4/Q5.
- **Q3 (Wave 3): line network coalescing** — glue touching
  same-class segments into strokes pre-assignment; feeds the
  existing simplify_coalesced_linestring path. Gate: roads become
  continuous lines at coarse zooms.
  ∥ **Q4: point clustering** (clustering.rs; needs Q6). Gate: NYC
  as graduated dots.
- **Q5 (Wave 4): polygon coalescing** — merge adjacent small
  polygons (unary union) when density demands; needs Q6, benefits
  from Q2; heaviest compute (pairs with V4 streaming). Gate:
  Moldova coarse levels. May prove unnecessary if Q2 suffices.

V3 benchmarks + E0 export-pmtiles do NOT depend on this ladder —
start them after Wave 1 passes the eyeball gate; they improve
automatically as waves land.

## Phase 3 — Ecosystem (parallel, after format freeze)

### E0. `gpq-tiles export-pmtiles`  (deps: P4, V2 quality gate;
reuses shelved mvt.rs/pmtiles_writer.rs/clip.rs)
DONE 2026-07-03: `overview/export.rs` + CLI subcommand; single-pass per-zoom
tiling (no external sort / budget retry), optional one-pass oversized valve.
Portland lines (z2–z14, 295k feats) → 46 MB PMTiles, 1325 tiles, 5.05s, 244 MB
RSS; tiles decode (tippecanoe-decode) with per-zoom totals matching the overview
report up to expected tile-border duplication (0%→7.2% z14). See
`benchmarks/overview/EXPORT_NOTES.md`.
(ADDED 2026-07-02.) Batch PMTiles export FROM an overview file —
the replacement for the stuck tile pipeline, not a revival of it.
Per zoom: read the matching level band (already thinned/simplified/
Hilbert-ordered), windowed sweep groups features into tiles, clip at
tile bounds, MVT-encode, write PMTiles. No global external sort, no
per-tile budget retry loop (generalization is precomputed); optional
light per-tile safety valve for pathological density. This gives the
immediate-adoption artifact (PMTiles works everywhere today) as a
cheap derived export of the canonical overview file — no drift.

### E1. `gpq-tiles serve`  (deps: P4; reuses shelved mvt.rs/clip.rs)
z/x/y MVT endpoint reading an overview file: level = zoom,
bbox = tile bounds, clip + encode precomputed features. Shares the
tile-window/clip/encode core with E0.
Gives every MapLibre map a zero-client-change path.
Stretch: static-deployable variant (Cloudflare Worker) later.

### E2. TS reader + demo  (deps: format freeze)
Separate repo. Thin loader (hyparquet or parquet-wasm):
footer → level select by viewport zoom → range requests →
GeoArrow/GeoJSON → deck.gl layer; MapLibre GeoJSON-source variant.
Hosted demo page streaming a large corpus file from R2/S3.
(cogp-js is the pattern reference.)

### E3. Python bindings + docs  (deps: P5)
pyo3 wrapper for convert/validate/read; DuckDB + QGIS + GeoPandas
recipes; user docs page.

### H1. Writer row-group pathology  (cheap; do first)
DONE 2026-07-03: per-level RG sizing + default stats suppression on
string/binary/WKB columns (`--full-column-stats` to opt out). Moldova dup
footer 8.84 → 0.24 MB, file 411 → 360 MB, regional viewport 37.6 → 21.8 MB,
street 10.8 → 2.0 MB; metros unregressed; all validate checks pass.
RESULTS.md revised in place (dated H1 note).

stats incl. ULID id strings, paid on every remote query) and 10k-row
RGs at coarse bands that defeat bbox pruning (regional viewport
overlaps ~all of them).

RGs (target: band ≤ N rows ⇒ single RG, else size by band share);
(b) disable statistics on high-cardinality string columns (id-like),
keep stats on bbox/level; (c) re-run bench_storage/bench_access for
Moldova and update RESULTS.md.
Acceptance: Moldova footer < 1 MB; regional-viewport bytes visibly
down; all validate checks still pass.

### H3. V4 streaming — the big-files unlock  (the big ticket)
Baseline to beat: Moldova 632k = 10m57s / 5.44 GB RSS (convert),
and export holds one full zoom in memory (canonical zoom = whole
dataset ⇒ same ceiling).
(a) Convert: pass 1 stream → winner tables (+ Q2 budget cuts);
pass 2 per level stream → simplify → write batch-by-batch. Memory
target O(row group + winner tables). The assign engine was built
for this (O(occupied-cells) state).
(b) Export: Hilbert order ⇒ tiles complete in sequence; flush
finished tiles instead of holding the zoom map.
(c) Wall-time profile while in there: Moldova is 3.6× slower than
tippecanoe — suspect per-level decode→rebuild churn (geometry
decoded/re-encoded once per level). Cache encoded WKB or cascade
simplification level-to-level.
Acceptance: Moldova convert < 1 GB RSS; planet-tier dataset
(--large Oregon roads, 5M feats) converts on a laptop; dhat/RSS
regression test.

### H4. Hostile-inputs wave  (correctness debt)
Known-unverified today:
- Hilbert-sorted input contract is ASSUMED, never checked — unsorted
  input silently produces terrible pruning. Add cheap sortedness
  sniff + loud warning (fallback sort = later, optional).
- Antimeridian: spec prohibits (§7.3) but converter likely does NOT
  reject — probable silent garbage bboxes. Enforce with clear error.
- High latitude: METERS_PER_DEGREE equatorial approximation ⇒
  thinning grids ~2.5× off in northern Norway. Decide: latitude-
  scaled factor or documented limitation; add a Norway/Svalbard
  corpus dataset either way.
- Empty/null geometries, mixed geometry-type columns, poles.
- GeoParquet 2.0 native-geometry inputs (spec §10 forward-compat is
  unimplemented in code).
Acceptance: each case either works with a test or fails with a
clear, documented error. Corpus grows hostile fixtures.

### H5. Quality ladder continuation (see Phase 2 §V5)
Order: Q6 aggregation semantics [HUMAN design decision] → Q4 point
clustering (owns remaining point storage overhead — NYC dup-mode is
+156% vs source; budget can't touch it, clustering can) → Q3 line
coalescing (coarse-zoom strokes; biggest remaining visual parity
gap) → Q5 polygon coalescing only if Moldova true-scale renders
demand it.

### H6. Parity & ecosystem tail
- Multi-layer support (issue #16): several inputs → one overview
  file / one tileset. Spec v0.2 design decision (layer column? per-
  layer level bands?). Real tippecanoe parity gap.
- E3 Python bindings for overview/validate/export-pmtiles.
- E1 serve; E2 TS loader + hosted demo (unblocks L2 blog).
- Remote-storage leg of V3 benchmarks (needs an R2/S3 bucket)
  before numbers go external.

Proposed sequence: H1 → H2 → H3 → H4 → (H5 ∥ H6), with L1 (the
one-pager to the GeoParquet maintainers) drafted after H1 fixes the
worst published number and sent whenever the human is ready — it
does not need to wait for H3+.
